# 统一实时数据接口技术规格书

## 目标
- 将默认分析数据改为实时接口获取，由统一数据管理器负责赛程、积分、伤停、球员进球与赔率的拉取、规范化、校验、溯源和快照。
- API-Football 作为赛程/积分/伤停/球员主数据源；The Odds API 作为世界杯1X2赔率源。
- 任何缺密钥、请求失败、数据过期、球队/时间无法匹配、赔率样本不足或必要字段缺失都必须失败停止，不得静默回退到旧JSON。

## 接口与安全约束
- `API_FOOTBALL_KEY` 仅通过请求头传递；`THE_ODDS_API_KEY` 按供应商要求通过查询参数传递。
- 密钥只从环境变量读取，不写入源码、日志、异常、快照或测试。
- HTTP仅允许HTTPS；设置超时、有限重试和明确User-Agent；禁止无限重试。
- 默认API-Football世界杯联赛ID为1、赛季为2026；均可用环境变量覆盖。
- 默认The Odds API赛事键为 `soccer_fifa_world_cup`，市场为 `h2h`，地区为 `us`。
- 赔率使用多家机构同一结果的中位数；严格模式至少需要2家机构，并保留机构数、更新时间和来源。
- 跨源赛事匹配必须同时满足球队名称规范化一致和开赛时间误差不超过15分钟。
- 世界杯淘汰规则为中央版本化权威配置，附FIFA来源；不调用未公开的FIFA私有接口。

## 统一数据输出
- 顶层必须包含 `data_as_of`、`sources`、`quality_checks`、`matches`。
- 每场必须包含统一赛程ID、双方球队ID/名称、开赛时间、阶段、赔率、球队上下文、规则和字段级溯源摘要。
- 伤停只表述为“接口当前已报告”，空列表不得解释为医学意义上的确定无伤。
- 可选将规范化结果保存为无密钥JSON快照，用于审计和严格时间截断回测。

## 实施步骤
1. 实现安全HTTP客户端，拒绝非HTTPS，超时并仅对临时错误有限重试，所有异常脱敏。
2. 实现两个供应商适配器及响应错误校验，不让供应商原始结构泄漏到分析器。
3. 实现统一数据管理器：获取当前可售世界杯赔率，匹配官方赛程主数据，再获取积分、伤停和球员进球。
4. 统一伤停状态、球队名称、开赛时间和赔率；执行数据完整性、新鲜度与跨源一致性检查。
5. 更新CLI：无 `--file` 时默认实时；`--file` 仅用于快照重放或回测；增加 `--doctor`、`--date`、`--save-snapshot`。
6. 用假接口覆盖成功和失败路径；无真实密钥时明确标记真实联调未执行。

## 验证方式
- `$env:PYTHONDONTWRITEBYTECODE='1'; python -m unittest discover -s tests -v`
- `python main.py --doctor`：当前无密钥时应给出缺失变量，不泄漏任何值。
- 使用测试假接口验证三场数据聚合、跨源匹配、中位数赔率和缺失数据失败。
- 配置真实密钥后：`python main.py --date 2026-06-29 --save-snapshot data\\snapshots\\2026-06-29.json`

## 回滚方案
- 恢复CLI默认本地文件入口；删除新增三个接口模块和接口测试，不影响已有分析器与历史回测文件。
